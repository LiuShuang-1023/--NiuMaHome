# v0.3.0 架构设计文档 - 地理粗筛 + SQLite 临时数据库 + 离线估算

> 状态：实施中
> 目标：从根本上解决 v0.2.x 的"通勤大量未测算"、"语义反向"、"缓存幽灵"三大顽疾

---

## 1. 核心理念

### 旧架构的本质问题（v0.2.2.x）

```
抓 200 条全量 → 拿前 30 条精算通勤 → 事后过滤
                ↑                       ↑
                高德 QPS=3，慢          没算的房源逃过过滤
```

**症状**：用户说"通勤≤20分钟"，列表里却出现 178 条"未测算"，里面混着 30/40/50 分钟的房子。

### 新架构（v0.3.0）

```
抓 200 条 → 目的地 geocode 1 次 → 房源批量 geocode → haversine 直线粗筛
        ↓
        粗筛后约 30-60 条候选（直线 ≤ 通勤半径）
        ↓
        全部跑【离线估算】（毫秒级，免费） → 立刻显示完整列表（带"📍估算"徽章）
        ↓
        用户对感兴趣的房源点【查询实时】 → 单条精算高德 → 更新徽章为"✓ 实时"
        ↓
        所有数据进 SQLite 临时DB（按 session_id 隔离）
        ↓
        排序/筛选/分页全在 DB 内完成（毫秒级）
        ↓
        会话超时（30分钟）或用户清空 → 物理删除
```

**优势**：
1. ✅ 列表里**所有房源都有通勤数据**（离线估算），符合"≤20min" 语义
2. ✅ 高德调用量从 30 次/搜索 → 5-10 次/搜索（只对用户点的房源）
3. ✅ SQLite 替代进程内字典，无幽灵 bug，调试友好
4. ✅ 排序/筛选 SQL 完成，毫秒响应

## 2. SQLite 表设计

数据库文件：`backend/data/sessions.db`（gitignore 排除）

### 2.1 sessions 表（会话）
```sql
CREATE TABLE sessions (
    session_id   TEXT PRIMARY KEY,    -- UUID4
    created_at   INTEGER NOT NULL,    -- Unix 时间戳
    last_active  INTEGER NOT NULL,    -- 每次请求自动更新
    requirement  TEXT NOT NULL,       -- ParsedRequirement JSON 序列化
    dest_lng     REAL,                -- 目的地经度（geocode 后填）
    dest_lat     REAL,                -- 目的地纬度
    dest_label   TEXT                 -- 目的地展示名
);
CREATE INDEX idx_sessions_active ON sessions(last_active);
```

### 2.2 listings 表（房源）
```sql
CREATE TABLE listings (
    session_id    TEXT NOT NULL,
    listing_id    TEXT NOT NULL,
    platform      TEXT NOT NULL,
    title         TEXT,
    community     TEXT,
    address       TEXT,
    url           TEXT,
    price_base    INTEGER,
    area          REAL,
    layout        TEXT,
    floor         TEXT,
    orientation   TEXT,
    rental_type_tag TEXT,
    image_url     TEXT,
    raw_json      TEXT,               -- 完整 Listing JSON
    geo_lng       REAL,               -- geocode 后的经度
    geo_lat       REAL,               -- 纬度
    geo_source    TEXT,               -- 'amap' / 'baidu' / NULL
    distance_km   REAL,               -- 直线距离目的地（haversine）
    is_filtered_out INTEGER DEFAULT 0,-- 1=被粗筛过滤掉（不展示）
    filter_reason TEXT,               -- 过滤原因（"距离过远 12.3km"）
    PRIMARY KEY (session_id, listing_id),
    FOREIGN KEY (session_id) REFERENCES sessions(session_id) ON DELETE CASCADE
);
CREATE INDEX idx_listings_session ON listings(session_id, distance_km);
```

### 2.3 commutes 表（通勤数据）
```sql
CREATE TABLE commutes (
    session_id     TEXT NOT NULL,
    listing_id     TEXT NOT NULL,
    source         TEXT NOT NULL,      -- 'offline' / 'amap' / 'baidu'
    best_duration_min INTEGER,
    transit_min    INTEGER,
    riding_min     INTEGER,
    walking_min    INTEGER,
    driving_min    INTEGER,
    transit_distance_km REAL,
    raw_json       TEXT,                -- 完整 CommuteSummary JSON
    computed_at    INTEGER NOT NULL,
    PRIMARY KEY (session_id, listing_id),
    FOREIGN KEY (session_id) REFERENCES sessions(session_id) ON DELETE CASCADE
);
```

### 2.4 costs 表（成本）
```sql
CREATE TABLE costs (
    session_id     TEXT NOT NULL,
    listing_id     TEXT NOT NULL,
    total          INTEGER NOT NULL,
    raw_json       TEXT NOT NULL,       -- 完整 CostBreakdown JSON
    PRIMARY KEY (session_id, listing_id),
    FOREIGN KEY (session_id) REFERENCES sessions(session_id) ON DELETE CASCADE
);
```

## 3. 离线估算算法

### 3.1 haversine 距离 → 各方式时长

基于实测的"距离-时长"经验系数（一二线城市路网迂回率 1.3-1.6 倍）：

| 出行方式 | 公式 | 备注 |
|---|---|---|
| 步行 | `直线km × 1.3 ÷ 5 km/h × 60` 分钟 | 5km/h 是均值步速 |
| 骑行 | `直线km × 1.4 ÷ 15 km/h × 60` 分钟 | 共享单车含红绿灯 |
| 公交 | `直线km × 1.6 ÷ 20 km/h × 60 + 8` 分钟 | +8min 等车换乘 |
| 驾车 | `直线km × 1.4 ÷ 30 km/h × 60` 分钟 | 市区均速 |

误差预估（基于广州/上海实测）：
- 短距离（<3km）：误差 ±15%
- 中距离（3-10km）：误差 ±25%
- 长距离（>10km）：误差 ±35%（因不知道是否有地铁直达）

### 3.2 粗筛阈值

用户指定"通勤 ≤ N 分钟"时，对应的最大直线距离：

| 通勤上限 | 最大直线半径（粗筛） |
|---|---|
| ≤ 20 min | 8 km |
| ≤ 30 min | 12 km |
| ≤ 45 min | 18 km |
| ≤ 60 min | 25 km |
| 不限 | 不粗筛 |

公式：`半径 km ≈ 通勤上限分钟 × 0.4`（保守估算，公交时速）

## 4. API 设计

### 4.1 改造的接口

#### POST `/api/search` （彻底重写）
**变化**：从"搜索 + 通勤"改为"搜索 + geocode + 粗筛 + 离线估算"

**请求**：
```json
{
  "session_id": "uuid-xxx",       // 新增：前端 sessionStorage 生成
  "requirement": {...},
  "sort_mode": "综合",
  "platform": "all",
  "page": 1,
  "page_size": 10
}
```

**响应**：与之前 SearchResponse 格式兼容 + 新增字段：
```json
{
  "recommendations": [...],         // 已粗筛 + 已估算
  "total_crawled": 200,
  "total_filtered": 60,             // 粗筛后剩余
  "geo_filter_stats": {             // 新增：粗筛统计
    "geocoded": 180,
    "geocode_failed": 20,
    "within_radius": 60,
    "out_of_radius": 120,
    "radius_km": 8.0
  },
  "precise_count": 0,               // 新增：已精算的数量
  ...
}
```

#### POST `/api/search/sort`（新接口，仅排序）
**用途**：用户切换排序模式时，**不重新爬不重新算**，直接 SQL ORDER BY

```json
{ "session_id": "xxx", "sort_mode": "通勤", "platform": "all", "page": 1 }
```

#### POST `/api/search/precise_one`（新接口）
**用途**：用户对单条房源点"查询实时"

```json
{ "session_id": "xxx", "listing_id": "lianjia_123" }
```
**行为**：
1. 调高德算精确通勤
2. 写入 `commutes` 表（覆盖离线数据，source='amap'）
3. 返回更新后的 Recommendation

#### POST `/api/search/precise_batch`（新接口，可选）
**用途**：批量精算前 N 条
```json
{ "session_id": "xxx", "max_count": 30 }
```

#### DELETE `/api/session/{session_id}`
**用途**：用户点"重新搜索"或关闭页面时清空数据

### 4.2 移除的接口

- `POST /api/search/recompute_commute` → 改用 `/precise_batch`
- `POST /api/search/reset_quota` → 保留（仍用得到）

## 5. 会话生命周期

### 5.1 创建
- 前端进入页面 → `sessionStorage.getItem('niumahome_session')`
- 没有则生成 UUID v4 → `sessionStorage.setItem(...)`
- 每次 API 请求都在 body 带 `session_id`

### 5.2 维持
- 后端每次处理请求 → 更新 `sessions.last_active`
- 关闭浏览器 → sessionStorage 自动清空
- 下次打开 → 新 session_id（旧数据等定时任务清理）

### 5.3 清理
- **定时任务**：FastAPI lifespan 启动后台 task，每 5 分钟扫一次
- **TTL**：`now - last_active > 30 min` → DELETE FROM sessions WHERE...
- 因 FK ON DELETE CASCADE，listings/commutes/costs 自动连带清空

### 5.4 用户主动清空
- 前端"重新搜索"按钮 → DELETE /api/session/{id}
- 前端不重新生成 session_id（继续用同一个）

## 6. 前端状态机

```
[初始] 生成 session_id → 存 sessionStorage
   ↓
[对话] 用户与 AI 聊天 → requirement 准备好
   ↓
[搜索中] POST /search → 显示 SearchingPanel
   ↓
[列表展示] 全部带"📍 估算"徽章 + 距离 km + 4 种通勤分钟数
   ↓ 用户操作分支：
   ├─ 切换排序/分页/平台 → POST /sort（毫秒响应）
   ├─ 点单条卡片"查询实时" → POST /precise_one → 该条徽章变"✓ 实时"
   ├─ 点"批量精算前 30 条" → POST /precise_batch
   └─ 点"重新搜索" → DELETE /session/{id} + 重新走流程
```
